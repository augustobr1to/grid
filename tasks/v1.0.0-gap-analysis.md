# Grid — what's missing for a 1.0.0 production engine

Assessment date 2026-06-29, branch `fix/council-critical-high`. State: build 6 projects green,
tests 28 (21 engine + 7 server integration), lint clean. CRITICAL multiplayer-correctness trio
and the HIGH leak/lifecycle set are fixed and verified. This document is the gap between *here*
and a defensible **1.0.0** tag for a production browser 3D game engine.

Legend: **[BLOCKER]** must ship in 1.0.0 · **[SHOULD]** strongly wanted · **[POST]** fine after 1.0.

---

## 1. Public API stability & type safety  — the biggest 1.0 gap
A 1.0.0 engine makes a stability promise about its public surface. Right now that surface is untyped.
- **[BLOCKER]** Engine still has 24+ `any` on the core host graph (`Game`, `Scene`, `GameObject`,
  `Component`, `Renderer`). No `ISceneHost` / `IGameHost` / `INetworkManager` interfaces — circular
  imports are worked around with `any`. The public API a consumer codes against is effectively untyped.
- **[BLOCKER]** `@typescript-eslint/no-explicit-any` is `off` workspace-wide — re-enable as `warn`→`error`
  and burn down.
- **[SHOULD]** `apps/game` is 13 untyped `.js` files; the reference game for the engine should be `.ts`
  so it exercises the typed API.
- **[SHOULD]** No documented "public vs internal" boundary (underscore-prefixed members like `_options`,
  `_rigidBody` are reached into from app code). Define the exported entrypoints and freeze them.

## 2. Test coverage & quality gates
- **[BLOCKER]** No coverage measurement or threshold anywhere; CI does not gate on coverage.
- **[BLOCKER]** Engine core largely untested: no tests for `Renderer` fixed-timestep accumulator,
  `PhysicsHelpers`, `RigidBodyComponent` sync, `KinematicCharacterController` ground/sweep, `AssetStore`
  cache/dispose, `ColyseusNetworkManager`. (Server room logic is now covered — 7 integration tests.)
- **[SHOULD]** No client/e2e test that actually boots the game against the server (Playwright-style)
  to prove the seed→buildWorld→sync→reconcile path in a browser.
- **[SHOULD]** No regression test for the nested rigid-body world/local bug (see §4).

## 3. Multiplayer completeness
- **[BLOCKER]** Client reconciliation is still hard-snap on X/Z (no input replay). `InputQueue` exists
  but is unused. Under real latency this rubber-bands. 1.0 needs prediction + server-reconciliation with
  input replay, or an explicit documented limitation.
- **[BLOCKER]** Server hardening: no input rate-limiting, no auth on join, unbounded room creation
  (`joinOrCreate` with no cap) = trivial memory DoS. Needs connection/message rate limits + room caps.
- **[SHOULD]** No area-of-interest / interest management — every client gets every player's full state;
  won't scale past a handful of players.
- **[SHOULD]** Reconnection is *surfaced* (connection-lost overlay) but there is no actual reconnect with
  backoff / state resync.
- **[SHOULD]** Voice runs on the public PeerJS broker (`0.peerjs.com`) — signaling leaves your infra;
  no mute / push-to-talk / per-peer block; answers any caller. Self-host PeerServer + add mute for 1.0.

## 4. Resource management correctness (engine)
- **[BLOCKER]** Nested rigid bodies are mis-positioned: body seeded from local position, world translation
  written back to child local — any non-root physics object is silently wrong. Needs world-space conversion.
- **[SHOULD]** `Game.dispose()` now exists, but scene-level lights and `SoundComponent` audio nodes are not
  disposed on unload; `ModelComponent.clone()` shares cached geometry/material so `AssetStore.unloadAll()`
  can corrupt live clones (needs ref-counting or SkeletonUtils clone).
- **[POST]** Two physics-stepping code paths (`Renderer.start` vs dead `Scene.update`) — remove the dead one.

## 5. Rendering
- **[SHOULD]** Verify the "Tron" composer (ACES + UnrealBloom) is actually active in the game's render path —
  the game uses unlit `MeshBasicMaterial`, so bloom depends on raw color brightness; confirm visually.
- **[SHOULD]** No WebGL context-loss handling (`webglcontextlost`/`restored`) — a lost context kills the app.
- **[POST]** `EnginePreview` polls `getBoundingClientRect()` every frame (layout reflow) — use ResizeObserver.

## 6. Build, release & distribution  — "1.0.0" implies a shippable artifact
- **[BLOCKER]** The engine is a workspace-only package — not published, no `npm`/CDN artifact, no install
  story. A 1.0.0 engine needs a real release (private registry or public npm) with provenance.
- **[BLOCKER]** size-limit gate is non-functional: threshold 800 kB vs actual ~139 kB (5.8× headroom, can
  never trip); the real app bundles (three + rapier) are ungated; the size step assumes engine `dist` is
  always built on `affected` PRs. Tighten to ~160 kB, gate app bundles, fix the build dependency.
- **[SHOULD]** No semver discipline: no CHANGELOG, no release workflow, no tag automation, no API-diff check
  (api-extractor is pinned but unused).
- **[SHOULD]** tsconfig src-vs-dist alias incoherence — `tsc` test/typecheck validates different artifacts
  per app than the build produces; adopt TS project references for ordered, incremental builds.

## 7. CI/CD & deployment
- **[BLOCKER]** No server deployment pipeline — the authoritative Colyseus server has no build-image /
  deploy / scaling story (Colyseus Cloud or container). A multiplayer engine isn't shippable without it.
- **[SHOULD]** CI does not gate coverage, has no e2e stage, runs a single Node version, and writes a fresh
  Nx cache every commit (`github.sha` in the primary key).
- **[SHOULD]** No `prettier --check` in CI (format only enforced locally); lint-staged lacks `--max-warnings=0`.

## 8. Documentation & examples
- **[BLOCKER]** No generated API docs (typedoc) and no "getting started / create your first scene" guide.
  A 1.0 engine needs a documented public API.
- **[SHOULD]** No runnable example projects beyond the one game; no migration/architecture guide for consumers.

## 9. Editor
- **[SHOULD]** The editor app is a non-functional shell: viewport is a 2D `<canvas>` placeholder, gizmo
  modes / grid / stats are stored in Redux but wired to nothing; Inspector shows stale values on selection
  switch (uncontrolled inputs). Either cut the editor from 1.0.0 scope explicitly, or make it engine-backed.

## 10. Observability, security & compat
- **[SHOULD]** Server has no structured logging, metrics, or error reporting beyond `/health`; no rate-limit
  or abuse protection (overlaps §3).
- **[SHOULD]** No browser/device support matrix; input is pointer-lock + keyboard only (no touch/mobile,
  no gamepad coherence — multi-pad handling is incoherent).
- **[POST]** Asset pipeline has no compression (Draco/KTX2) or streaming; fine for a small map, not for scale.

## 11. Dead/unimplemented gameplay systems (reference game)
- **[SHOULD]** Scoreboard never updated, CapturePoint visuals never refreshed, audio (SFXManifest) has no
  playback, "RETURN TO LOBBY" button is a dead end. Either implement or remove before calling the game a
  1.0 showcase.

---

## Recommended 1.0.0 cut line
**Must-have blockers (engine credibility):** §1 typed public API + interfaces · §2 coverage gate + engine-core
tests · §3 input-replay reconciliation + server rate-limit/room-cap · §4 nested rigid-body fix · §6 real
published artifact + working size gate · §7 server deploy pipeline · §8 API docs + getting-started.

**Defensible to defer to 1.x:** AOI/interest management, self-hosted voice + mute, editor (cut or label
"preview"), asset compression/streaming, mobile/touch, observability stack.

**Rough effort:** the blockers are ~2–3 focused waves (types+tests, multiplayer hardening, release/deploy/docs).
The already-landed CRITICAL/HIGH fixes cleared the "is it correct" bar; the remaining 1.0 work is mostly
"is it stable, shippable, and documented."
